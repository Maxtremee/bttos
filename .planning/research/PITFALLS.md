# Domain Pitfalls: Twitch webOS Alt

**Domain:** Alternative Twitch client for LG webOS TV
**Researched:** 2026-04-14
**Overall confidence:** MEDIUM-HIGH (most claims verified against official docs or multiple community sources)

---

## Critical Pitfalls

Mistakes that cause rewrites, blocked apps, or unrecoverable runtime failures.

---

### Pitfall 1: Twitch Stream URL Extraction Is Fragile by Design

**What goes wrong:** Twitch does not expose direct HLS stream URLs through the official API. Getting a playable URL requires calling the internal `PlaybackAccessToken` GQL mutation (at `https://gql.twitch.tv/gql`) to obtain a signed token, then passing it to the Usher service (`https://usher.twitchnw.tv/api/channel/hls/{channel}.m3u8`). Both endpoints are undocumented and unsupported for third parties.

**Why it happens:** Twitch's web player negotiates a cryptographic access token using client-integrity (CI) tokens generated by obfuscated JavaScript running in a browser. The CI token is calculated in-browser and is effectively impossible to reverse-engineer in a standalone app. When Twitch enforces CI token requirements (which it has done intermittently since 2023), any client that cannot produce a valid CI token is refused a stream URL.

**Consequences:** The stream playback feature — the core of this app — can stop working without warning at any time due to a server-side Twitch policy change, with no official recourse.

**Prevention:**
- Use the official OAuth user token when requesting PlaybackAccessToken to reduce the likelihood of CI enforcement. Authenticated requests are treated differently than anonymous ones.
- Design the stream URL fetching layer as a thin, isolated module so it can be patched independently of the rest of the app.
- Monitor open-source clients (Streamlink's twitch.py, youtube-webos) for workarounds when Twitch deploys new restrictions.
- Accept this as an intentional constraint of building against unofficial endpoints — plan for it to break.

**Warning signs:** 403 or 401 responses from the GQL endpoint or Usher; "access token failed" errors; open GitHub issues in Streamlink against the twitch plugin.

**Phase:** Address in the video playback milestone. Document the architecture decision explicitly so future maintainers understand why the extraction layer must remain swappable.

### Pitfall 2: Simulator Works, Real TV Hardware Fails

**What goes wrong:** The webOS TV Simulator uses a modern Chromium build, but real TV hardware runs a frozen, much older Chromium version. webOS 4.x runs Chromium 53 (2016). webOS 5.x runs Chromium 68 (2018). webOS 22/23 runs approximately Chromium 87 (2020). An app that passes all simulator tests can silently fail on the target device due to missing JavaScript features, CSS properties, or Web APIs.

**Why it happens:** LG ships a static browser engine per webOS version. The version never updates unless LG pushes a firmware upgrade, which varies by TV model year.

**Consequences:** ES2020+ syntax (optional chaining, nullish coalescing, logical assignment) may fail on older targets. Dynamic `import()`, CSS `gap` in flexbox, and `IntersectionObserver` are absent on older webOS. Build output using modern module syntax (ESM) without transpilation will fail.

**Prevention:**
- Transpile all JavaScript to ES5 with Babel or use esbuild with a `--target=chrome68` flag to cover the likely device range.
- Test on a real TV early — do not rely on the simulator for compatibility verification.
- Set `browserslist` targets to `Chrome >= 68` and validate with `eslint-plugin-compat`.
- Note: webOS TVs from 2021 onward running webOS 6+ are more lenient, but targeting Chrome 68+ is the safe floor.

**Warning signs:** App works in simulator but blank screen on TV; uncaught SyntaxError in ares-inspect logs; features working on newer TV but failing on 2019-era models.

**Phase:** First phase — establish build pipeline with correct transpilation before writing any feature code.

### Pitfall 3: HLS.js Freezes and Memory Exhaustion During Live Streams

**What goes wrong:** HLS.js has known bugs on webOS LG TVs where live stream playback freezes after 5-10 minutes. Multiple recovery attempts by HLS.js can exhaust the device's limited RAM. Once memory is full, the Chromium process in webOS is killed and the app crashes.

**Why it happens:** The webOS MSE (Media Source Extension) implementation has quirks that differ from the spec. HLS.js does not officially test against webOS. Buffer management assumptions in HLS.js do not account for the slow garbage collection and limited memory (~256MB-512MB heap in older devices) of TV hardware.

**Consequences:** Live streams — the entire purpose of this app — crash within minutes of use on lower-end hardware.

**Prevention:**
- Configure HLS.js with aggressive buffer limits: `maxBufferLength: 15`, `maxMaxBufferLength: 30` (seconds), `backBufferLength: 0` to prevent growing the backward buffer.
- Set `maxFragLookUpTolerance` and tune `abrEwmaFastLive` for TV-hardware packet rates.
- Set up a periodic buffer health check and proactively flush the back buffer while the stream is playing.
- Consider THEOplayer (commercial) or Shaka Player as alternatives if HLS.js proves unusable; both have documented webOS support.
- Never rely on HLS.js recovery loops — catch error events and implement a manual hard-reload of the `<video>` element instead.

**Warning signs:** Memory climbing in ares-inspect's performance tab; stream freezing at consistent intervals; `BUFFER_STALLED_ERROR` events in HLS.js error callbacks.

**Phase:** Video playback milestone. Profile memory during sustained viewing (30+ minutes) before shipping.

## Moderate Pitfalls

Mistakes that degrade UX significantly or require non-trivial rework.

---

### Pitfall 4: Focus Management After Route or Modal Transitions

**What goes wrong:** When navigating between screens (channel list → playback → chat overlay) or opening/closing overlays, the focused element is lost or reset to the first focusable element on the page. The user's D-pad input then lands on the wrong element.

**Why it happens:** DOM re-renders triggered by SolidJS reactive updates or route changes tear down and recreate elements. The browser's native focus state does not survive element recreation. There is no automatic spatial navigation in a plain web app — it must be coded explicitly.

**Prevention:**
- Implement a focus management singleton that tracks the last focused element ID per "screen" and restores it on return.
- Use `js-spatial-navigation` (luke-chang) or `@noriginmedia/norigin-spatial-navigation` adapted for SolidJS signals.
- Every modal, overlay, and route transition must explicitly set focus on mount and restore it on unmount.
- Test the full navigation flow: channel list → select channel → playback → open chat overlay → close overlay → D-pad back to channel list. Each step must land focus exactly where a user expects.

**Warning signs:** Press Back and focus jumps to the top of a list instead of the previously selected item; overlay closes but no element is focused; remote stops responding because a non-focusable element has focus.

**Phase:** Foundation phase (navigation primitives) and must be re-validated at every feature addition.

### Pitfall 5: Chat Message Flood Causing Render Jank

**What goes wrong:** Popular Twitch channels can deliver 50-200+ chat messages per second. Naively rendering each message as a reactive DOM update in SolidJS will generate hundreds of DOM mutations per second, stalling the render thread on TV hardware and causing visible frame drops in the video overlay.

**Why it happens:** Even with SolidJS's fine-grained reactivity, inserting DOM nodes at high frequency overwhelms the browser's compositor. Each message with emotes triggers image decode and layout. WebOS Chromium 68-87 has slower GPU compositing than modern desktop browsers.

**Prevention:**
- Batch incoming messages: collect messages for 100-200ms intervals and append in bulk rather than one-by-one.
- Cap the DOM to the last N messages (e.g., 50-100 messages max); prune older entries on each batch insert.
- Use a `<For>` with a fixed-length derived signal instead of a growing list — let SolidJS diff only the new tail.
- Lazy-load emote images: use `loading="lazy"` and provide fixed dimensions to prevent layout reflow per emote.
- Provide a chat toggle (already planned in requirements) that completely unmounts the chat component tree — do not use `display: none` which still processes IRC messages in the background.

**Warning signs:** Video stutters when chat is open but smooth when closed; DevTools shows long task durations correlating with chat updates; memory growing linearly over time with chat open.

**Phase:** Chat rendering milestone. Load-test against a large Twitch channel (e.g., >10k concurrent viewers) before considering this milestone complete.

### Pitfall 6: Twitch OAuth Refresh Token Mishandled

**What goes wrong:** Twitch refresh tokens are single-use. If the app calls the refresh endpoint while a previous refresh is still in-flight (e.g., multiple requests triggered simultaneously), the first response invalidates the token for subsequent calls. Any call using the old token gets a 401, the app detects "needs refresh," triggers another refresh — and enters an infinite 401 loop, logging the user out.

**Why it happens:** TV apps are long-running sessions (days between reboots). Access tokens expire every 4 hours. If multiple background processes or tab wake-ups trigger concurrent refresh calls, the race condition fires.

**Prevention:**
- Implement a token refresh singleton with a promise deduplication pattern: if a refresh is already in flight, return the same promise to all callers rather than issuing a new request.
- Persist both the new access token and the new refresh token atomically (localStorage in webOS) on every successful refresh. A crash mid-write must not corrupt both values — write the new refresh token last.
- On a 401 from a refresh attempt, redirect immediately to the device code auth flow rather than retrying.
- Store token expiry timestamp and proactively refresh 5 minutes before expiry rather than reactively refreshing on 401.

**Warning signs:** User is unexpectedly logged out after the TV wakes from standby; 401 errors in network log followed by a cascade of additional 401s; two simultaneous requests to `/token` endpoint in the same second.

**Phase:** Authentication milestone. The refresh singleton must be in place before building any API calls on top of it.

### Pitfall 7: Twitch Embedded Ads Causing Black Screen or Discontinuity

**What goes wrong:** Twitch injects server-side ads directly into the HLS stream segments. During an ad break, the stream emits an `EXT-X-DISCONTINUITY` tag and switches to a different video track. HLS.js may pause output, stall, or show a black screen during these transitions. On webOS, the `EXT-X-DISCONTINUITY` support in the MSE layer is incomplete.

**Why it happens:** Twitch's SureStream (SSAI) stitches ads at the server before delivery. Third-party clients receive these segments without advance notice. The codec parameters or resolution can change at the discontinuity boundary, confusing some MSE implementations.

**Prevention:**
- When using HLS.js, subscribe to the `FRAG_CHANGED` event and detect discontinuity flags. Display a "Stream loading..." indicator rather than showing a frozen black frame.
- Test ad break handling explicitly; do not assume the player recovers gracefully.
- Note: complete ad suppression is not achievable through a third-party client — Twitch continuously patches workarounds. Do not build the app's value proposition around being "ad-free."

**Warning signs:** Black screen for 15-30 seconds periodically; stream freezing at round-clock intervals (every 20-30 minutes, which is typical Twitch ad cadence); `BUFFER_STALLED_ERROR` correlating with ad breaks.

**Phase:** Video playback milestone. This is a known ongoing issue — document it as a known limitation rather than a bug to fix.

### Pitfall 8: webOS Developer Mode Session Expiry Removes App

**What goes wrong:** When sideloading the app in developer mode (not distributed through the LG store), the developer mode session expires after a configurable period (999 hours max). When it expires, LG removes all developer-installed IPK apps from the TV. The TV user has no app anymore and must re-sideload.

**Why it happens:** LG's developer mode is designed for app testing, not long-term deployment. The 999-hour timer (approximately 42 days) is a hard limit. The key server passphrase used for `ares-setup-device` can also reset independently of the timer.

**Prevention:**
- Set a calendar reminder to renew developer mode before the session expires.
- Build a one-command re-sideload script (`npm run deploy` calling `ares-package` then `ares-install`) so re-installation is fast.
- Document the renewal process in the repository README for the user's reference.
- For permanent installation, explore the webOS homebrew community's Homebrew Channel as an alternative distribution path that does not rely on developer mode sessions.

**Warning signs:** App disappears from TV home screen without user action; `ares-install` fails with authentication errors; Developer Mode app shows expired session.

**Phase:** Deployment and distribution milestone. Address before first stable sideload.

## Minor Pitfalls

Issues that cause friction but are easy to fix once identified.

---

### Pitfall 9: SolidJS Prop Destructuring Loses Reactivity

**What goes wrong:** Destructuring props in a SolidJS component (`const { title, count } = props`) converts reactive signals into plain values captured at call time. The component will not update when props change.

**Prevention:** Always access SolidJS props as `props.title`, `props.count`, etc. Use `splitProps` from `solid-js` when you need to separate groups of props. This is a SolidJS-specific rule and must be enforced via linting or code review.

**Phase:** Foundation phase — establish this convention before writing feature components.

---

### Pitfall 10: appinfo.json ID Mismatch Silently Breaks Packaging

**What goes wrong:** The `id` field in `appinfo.json` must be a reverse-domain-style string (e.g., `com.example.twitchalt`). If the app ID in `appinfo.json` does not match the directory structure expected by `ares-package`, or if the version field is out of sync between `package.json` and `appinfo.json`, packaging silently produces a malformed IPK that installs but immediately crashes.

**Prevention:**
- Keep a single source of truth: set `version` in `package.json` and have the build script write it into `appinfo.json` before packaging.
- Validate appinfo.json in CI using a JSON schema.
- The `id` must remain stable — changing it after first install creates a new app entry rather than upgrading the existing one.

**Phase:** Build pipeline setup in the foundation phase.

---

### Pitfall 11: Magic Remote Pointer and D-Pad Navigation Coexist and Conflict

**What goes wrong:** LG's Magic Remote can operate as either a pointer (laser cursor) or a D-pad. Users switch between modes dynamically by pressing directional keys (which activates D-pad mode) or moving the remote (which activates pointer mode). If the app only handles one mode, interaction breaks in the other.

**Prevention:**
- Ensure all focusable elements are large enough to tap with the pointer cursor (min 44x44px logical pixels).
- All hover styles must also apply to the `:focus` state — if information is only visible on hover, D-pad users can never see it.
- Use `cursor: default` styling to ensure pointer interactions feel native.
- Test both navigation modes explicitly.

**Phase:** UI/UX milestone and re-validated throughout.

---

### Pitfall 12: Twitch IRC Emote Positions Are Byte-Based, Not Character-Based

**What goes wrong:** The Twitch IRC `PRIVMSG` message format specifies emote positions using byte offsets into the message string, not character indices. Multibyte Unicode characters (emoji, accented characters) cause emote offset calculations to be off by one or more positions if you use standard JavaScript string indexing.

**Prevention:**
- When parsing emote positions from the `emotes` tag in `PRIVMSG`, encode the message to UTF-8, compute the byte offsets, and then decode back to characters.
- Several open-source Twitch chat parsers handle this correctly (`@kararty/dank-twitch-irc`). Prefer using a proven parser over rolling custom IRC parsing.

**Phase:** Chat rendering milestone.
